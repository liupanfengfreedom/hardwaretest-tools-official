📖 紹介:

デュアルモード テスト

ICMP Ping (リモート プロキシ) と HTTPS GET (ブラウザ ダイレクト) を並列して使用し、さまざまな診断ニーズに対応します。

プロキシ Ping はブラウザの制限を回避します。 HTTPS は実際のユーザー エクスペリエンスを直接反映します。ネットワーク層とアプリケーション層の両方からネットワークを分析します。DevOps、Web サイトの監視、ゲームの遅延分析に最適です。

リアルタイム可視化

レイテンシーの変化、色分けされた成功/失敗リクエストの動的な折れ線グラフ。

グラフは自動スケールされ、異常が強調表示されます。遅延のピーク期間を素早く特定し、ネットワークを最適化します。

スマート履歴

最近テストしたターゲットを自動的に保存します。入力フィールドをクリックして選択し、ワンクリックでクリアします。

履歴はローカルに保存され、アップロードされることはありません。頻繁に使用するターゲットを再利用し、生産性を向上させます。

端末ログ

各リクエストのシーケンス、応答時間、ステータス (HTTPS)、または損失の表示が明確に印刷されます。

最大 30 エントリ、スクロール可能。あらゆる詳細をトレースして、断続的な問題を診断します。

ジッター計算

連続する遅延間の平均絶対差を自動的に計算します。これは正確な安定性の指標です。

ジッター >30ms は、リアルタイム通信 (VoIP、ビデオ) に影響を与える可能性があります。ジッターを監視してネットワーク調整をガイドします。

連続モード

「継続」を選択すると、手動で停止するまでリクエストが送信されます。長期的な安定性の監視に最適です。

中断するには、「テストの停止」(開始後に赤色に変わります) をクリックします。変動期を捉えるのに最適です。

よくある質問 (FAQ)

自分のサーバーまたは Web サイトをテストするにはどうすればよいですか?

ターゲット フィールドにドメインまたは IP (例: example.com または 93.184.216.34) を入力し、テスト モード (Ping または HTTPS) を選択して、[開始] をクリックします。 HTTPS テストの場合は、サーバーがポート 443 をリッスンしていることを確認してください。

Ping と HTTPS の遅延が異なるのはなぜですか?

ICMP Ping はネットワーク層のラウンドトリップ時間を測定しますが、HTTPS GET には TCP ハンドシェイク、TLS ネゴシエーション、HTTP リクエスト/レスポンスが含まれます。これには通常、予想通り数十ミリ秒長くかかります。

異常とみなされるパケット損失率はどれくらいですか?

有線ネットワークの場合、ほとんどの場合、損失は 0% になります。 Wi‑Fi の場合、最大 0.1% まで許容されます。一貫して 0.5% を超える損失やタイムアウトが頻繁に発生する場合は、重大な問題があることを示しています。ローカル ネットワークを確認するか、ISP に問い合わせてください。

Ping にリモート プロキシを使用するのはなぜですか?

セキュリティ制限のため、ブラウザは生の ICMP パケットを送信できません。このツールはバックエンド プロキシ (またはパブリック API) を使用して Ping を実行し、結果を返すため、任意のターゲットに対する ICMP テストが可能になります。

プロキシ サーバーの場所は Ping 結果に影響しますか?

はい – ICMP Ping 遅延には、現在地からではなく、プロキシからターゲットまでの時間が含まれます。ただし、損失とジッターは依然としてターゲットのネットワークの安定性を反映しています。 HTTPS GET はブラウザから直接実行されるため、影響を受けません。

HTTPS で「ローカル ブロック」が表示されることがあるのはなぜですか?

リクエストが 5 ミリ秒以内に失敗した場合、通常はブラウザ拡張機能 (広告ブロッカーなど) によってブロックされます。このツールは、ネットワーク障害ではなく「ローカル ブロック」としてマークします。拡張機能を無効にして再テストしてみてください。

連続モードではどのくらいのデータ量が使用されますか?

各 Ping リクエストはわずか数十バイトです。 HTTPS リクエストは軽量です (HEAD/no-cors)。 1 時間後でも、合計トラフィックは通常 10MB 未満なので、安全に使用できます。
