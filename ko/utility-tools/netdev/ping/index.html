듀얼 모드 테스트

ICMP Ping(원격 프록시)과 HTTPS GET(브라우저 직접)이 나란히 있어 다양한 진단 요구 사항을 충족합니다.

프록시 Ping은 브라우저 제한을 우회합니다. HTTPS 다이렉트는 실제 사용자 경험을 반영합니다. 네트워크 계층과 애플리케이션 계층 모두에서 네트워크를 분석합니다. DevOps, 웹사이트 모니터링, 게임 대기 시간 분석에 이상적입니다.

실시간 시각화

지연 시간 변화, 색상으로 구분된 성공/실패 요청의 동적 선 차트입니다.

차트가 자동으로 조정되고 이상 항목이 강조 표시됩니다. 최대 지연 시간을 빠르게 파악하고 네트워크를 최적화하세요.

스마트 히스토리

최근 테스트한 대상을 자동으로 저장합니다. 입력 필드를 클릭하여 하나를 선택하고 원클릭으로 지웁니다.

기록은 로컬에 저장되며 업로드되지 않습니다. 자주 사용하는 대상을 재사용하여 생산성을 높이세요.

터미널 로그

각 요청의 순서, 응답 시간, 상태(HTTPS) 또는 손실 표시가 명확하게 인쇄됩니다.

최대 30개 항목, 스크롤 가능. 간헐적인 문제를 진단하기 위해 모든 세부 사항을 추적합니다.

지터 계산

연속 지연 간의 평균 절대 차이를 자동으로 계산합니다(정확한 안정성 지표).

지터 >30ms는 실시간 통신(VoIP, 비디오)에 영향을 줄 수 있습니다. 지터를 모니터링하여 네트워크 조정을 안내합니다.

연속 모드

수동으로 중지될 때까지 요청을 보내려면 "계속"을 선택하세요. 이는 장기적인 안정성 모니터링에 이상적입니다.

중단하려면 "테스트 중지"(시작 후 빨간색으로 변함)를 클릭하세요. 변동 기간을 포착하는 데 적합합니다.

자주 묻는 질문(FAQ)

내 서버나 웹사이트를 어떻게 테스트하나요?

대상 필드에 도메인 또는 IP(예: example.com 또는 93.184.216.34)를 입력하고 테스트 모드(Ping 또는 HTTPS)를 선택한 후 시작을 클릭합니다. HTTPS 테스트의 경우 서버가 포트 443에서 수신 대기하는지 확인하세요.

Ping 지연 시간과 HTTPS 지연 시간이 다른 이유는 무엇입니까?

ICMP Ping은 네트워크 계층 왕복 시간을 측정하는 반면, HTTPS GET에는 TCP 핸드셰이크, TLS 협상 및 HTTP 요청/응답이 포함됩니다. 이는 일반적으로 예상되는 대로 수십 밀리초 더 오래 걸립니다.

비정상적인 것으로 간주되는 패킷 손실률은 얼마입니까?

유선 네트워크의 경우 손실은 대부분 0%여야 합니다. Wi-Fi의 경우 최대 0.1%가 허용됩니다. 0.5%를 초과하는 지속적인 손실 또는 빈번한 시간 초과는 심각한 문제를 나타냅니다. 로컬 네트워크를 확인하거나 ISP에 문의하십시오.

Ping에 원격 프록시를 사용하는 이유는 무엇입니까?

브라우저는 보안 제한으로 인해 원시 ICMP 패킷을 보낼 수 없습니다. 이 도구는 백엔드 프록시(또는 공개 API)를 사용하여 Ping을 수행하고 결과를 반환하므로 모든 대상에 대한 ICMP 테스트가 가능합니다.

프록시 서버 위치가 Ping 결과에 영향을 미치나요?

예 – ICMP Ping 대기 시간에는 사용자 위치가 아닌 프록시에서 대상까지의 시간이 포함됩니다. 그러나 손실과 지터는 여전히 대상의 네트워크 안정성을 반영합니다. HTTPS GET은 브라우저에서 직접 실행되므로 영향을 받지 않습니다.

HTTPS에 가끔 '로컬 차단'이 표시되는 이유는 무엇인가요?

요청이 5ms 이내에 실패하면 일반적으로 브라우저 확장 프로그램(예: 광고 차단기)에 의해 차단됩니다. 도구는 이를 네트워크 장애가 아닌 "로컬 블록"으로 표시합니다. 확장 프로그램을 비활성화하고 다시 테스트해보세요.

연속 모드는 데이터를 얼마나 사용하나요?

각 Ping 요청은 수십 바이트에 불과합니다. HTTPS 요청은 가볍습니다(HEAD/no-cors). 한 시간이 지난 후에도 총 트래픽은 일반적으로 10MB 미만이므로 사용하기에 안전합니다.
